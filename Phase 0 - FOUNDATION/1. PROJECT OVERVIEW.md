
**Middleware Modes:**

**MULTI Mode (`APP_MODE=MULTI`):**

- Detects tenant via subdomain (apple.saas.com)
- Loads tenant from database
- Injects `tenantId` into request context
- All queries auto-filter by tenant

**SINGLE Mode (`APP_MODE=SINGLE`):**

- Ignores subdomain
- Uses fixed `DEFAULT_TENANT_ID=1`
- All queries use same tenant (transparent)
- UI hides tenant creation

**Key Features:**

- **Tenant-scoped users:** Regular SaaS users (99% of cases)
- **Tenant-less users:** Super Admins, onboarding users, platform managers
- **Multi-tenant users:** Can belong to multiple tenants, switch context
- **Row-Level Security:** Prisma middleware auto-injects `WHERE tenantId = ?`

**Upgrade Path:**

```bash
# Start as single-tenant
APP_MODE=SINGLE

# Years later, customer wants multi-tenant
APP_MODE=MULTI

# No code changes needed!
```

### 2. Extensible Schema Philosophy

Every Prisma model is designed to be **extended, not replaced**.

**The Cloneable Pattern:**

```prisma
// âœ… KAVEN BASE (boilerplate)
model User {
  id        String   @id @default(uuid())
  email     String   @unique
  name      String
  tenantId  String?  // Nullable = allows tenant-less users
  metadata  Json?    // Extensibility for custom fields
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  
  tenant    Tenant?  @relation(fields: [tenantId], references: [id])
}

// âœ… YOUR SAAS (after cloning)
model User {
  // Keep all base fields
  id        String   @id @default(uuid())
  email     String   @unique
  name      String
  tenantId  String?
  metadata  Json?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  
  // Add custom fields
  avatar    String?
  bio       String?
  timezone  String   @default("America/Sao_Paulo")
  preferences Json?
  
  // Add custom relations
  tenant    Tenant?  @relation(fields: [tenantId], references: [id])
  posts     Post[]   // Your custom model
}
```

**Extension Strategies:**

1. **Core Fields:** Never remove (id, email, tenantId, createdAt, updatedAt)
2. **Custom Fields:** Add specific to your SaaS (avatar, bio, settings)
3. **Metadata JSON:** For highly dynamic data (preferences, config)
4. **Custom Relations:** Link to your own models (User â†’ CustomModel)

**Why This Matters:**

- âœ… Updates from Kaven won't break your schema
- âœ… You can pull security fixes to core fields
- âœ… Your customizations stay isolated
- âœ… Multi-tenant isolation works regardless of extensions

## ğŸ”„ VERSIONING & UPDATE STRATEGY

### The Boilerplate Dilemma

**Classic Problem:**

1. Clone boilerplate â†’ Customize heavily
2. Boilerplate releases security patch
3. How to apply update without losing customizations?

**Kaven's Solution: 3-Tier Strategy**

---

### **Tier 1: Git Upstream (Simple, MVP)**

For developers who want full control:

```bash
# Setup
git clone https://github.com/kaven/boilerplate my-saas
cd my-saas
git remote add upstream https://github.com/kaven/boilerplate

# Update months later
git fetch upstream
git merge upstream/main --allow-unrelated-histories
# Resolve conflicts manually
```

**âœ… Pros:** Full ownership, no abstraction  
**âŒ Cons:** Merge conflicts, manual work  
**Use when:** MVP stage, learning Kaven, small customizations

---

### **Tier 2: NPM Packages (Recommended, POST-MVP)**

Core features as versioned packages:

```bash
# Create new SaaS
npx create-kaven-app my-saas
cd my-saas

# Install modules
npm install @kaven/core @kaven/payments @kaven/crm

# Update later (semantic versioning)
npm update @kaven/core   # Gets bug fixes (1.2.3 â†’ 1.2.4)
npm install @kaven/core@2.0.0  # Explicit major upgrade
```

**âœ… Pros:** Clean updates, modular, testable  
**âŒ Cons:** More setup, requires npm registry  
**Use when:** Production SaaS, team collaboration, multiple projects

---

### **Tier 3: Hybrid (Best of Both)**

Structure via Git + Features via Packages:

```bash
# Clone base structure (Docker, CI/CD, configs)
git clone https://github.com/kaven/boilerplate my-saas

# Install feature packages
npm install @kaven/core @kaven/payments

# Update structure (manual merge)
git fetch upstream && git merge upstream/main

# Update features (automatic)
npm update
```

**âœ… Pros:** Flexibility + convenience  
**âŒ Cons:** Two update mechanisms  
**Use when:** Need structure customization + feature stability

---

### **Recommended Migration Path**

1. **MVP (Months 1-2):** Use Tier 1 (Git clone)
2. **v1.0 (Month 3):** Migrate to Tier 2 (extract packages)
3. **Production:** Tier 3 (structure fork + packages)
